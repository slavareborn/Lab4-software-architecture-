# Лабораторна робота 4: Аналіз синхронної та асинхронної комунікації

## Що змінилося порівняно з лабораторною 3?
У попередній лабораторній роботі логіка виконувалася "в лінію". Всі дії знаходилися в межах одного Handler-а. У цій роботі було виділено побічні ефекти в окремі компоненти та розірвано жорстку зв'язність між основною бізнес-логікою та допоміжними системами.

## Виділені побічні операції
**Побічна операція:** Відправка email-нотифікації користувачу після успішної видачі книги.
**Компонент:** `NotificationService`. Він винесений в окремий модуль `infrastructure/notifications` і реалізує інтерфейс `INotificationService` для збереження принципу інверсії залежностей (DIP).

## Порівняння синхронного та асинхронного підходів

| Критерій | Синхронна комунікація (`BorrowBookHandlerSync`) | Асинхронна комунікація (`BorrowBookHandler`) |
| :--- | :--- | :--- |
| **Час відповіді API** | Збільшується. Клієнт змушений чекати, поки відправиться email (додатково 1-2 секунди). | Мінімальний. API повертає відповідь одразу після збереження в БД та публікації події в EventBus. |
| **Поведінка при збоях** | Якщо `NotificationService` падає, помилку потрібно перехоплювати (`try-catch`), інакше відкотиться збереження книги. | Збій при відправці листа ніяк не впливає на основну транзакцію видачі книги. |
| **Зв'язаність (Coupling)** | Середня. Handler напряму викликає сервіс нотифікацій. Якщо додасться ще аудит, доведеться змінювати Handler. | Низька. Handler нічого не знає про те, хто підписаний на подію `BookBorrowedEvent`. |
| **Складність реалізації**| Низька. Простий виклик методу через інтерфейс. | Вища. Потрібно реалізувати EventBus, класи подій (Events) та підписників (Subscribers). |

## Вибір для production-системи
Для production-системи я б однозначно обрав **асинхронний підхід**. 
Відправка нотифікацій, генерація звітів чи аналітика — це побічні процеси, які не повинні блокувати користувача. Крім того, в реальній системі замість in-memory `EventBus` варто використати зовнішній брокер повідомлень (наприклад, RabbitMQ або Kafka). Це забезпечить гарантовану доставку подій: якщо сервіс нотифікацій тимчасово впаде, повідомлення залишиться в черзі брокера і буде опрацьовано після перезапуску сервісу.

## Ідемпотентність та дублювання подій
**Що станеться, якщо одна подія буде доставлена двічі?**
У поточній реалізації підписник відправить два однакових листи користувачу.

**Чи ідемпотентні обробники?**
Наразі ні. Щоб зробити обробник ідемпотентним, у події `BookBorrowedEvent` було додано унікальний `eventId`. В production-середовищі Subscriber повинен перевіряти в базі даних або кеші (Redis), чи оброблявся вже запис з таким `eventId`. Якщо так — ігнорувати подію.
